# 智能仓储管理系统 — 五人分工方案（1 老手 + 4 菜鸟）

## 角色分配

| 角色 | 成果 | 代码量 | 定位 |
|------|------|--------|------|
| 老手 | 3 个模块 + 全部头文件 + 最终集成 | 大 | 算法核心，技术兜底 |
| 菜鸟 A | 2 个模块 | 中 | 复制粘贴型 |
| 菜鸟 B | 1 个模块 | 中 | 简单循环函数 |
| 菜鸟 C | 2 个模块 | 大 | I/O + UI（代码量最大，但简单） |
| 菜鸟 D | 1 行代码 + 文档 | 极少 | 文档主力，最菜的一个 |

---

## 老手：算法核心 + 全组接口 + 最终集成

| 文件 | 说明 |
|------|------|
| `Common.h` | 全局常量（全组讨论，老手维护） |
| `Goods.h` | 货物结构体 |
| `CategoryTree.h` | BST+旁路 类别索引（头文件） |
| `CategoryTree.cpp` | BST 改编，项目最难的设计点 |
| `MatGraph.h` | 邻接矩阵图（头文件） |
| `MatGraph.cpp` | Dijkstra + Floyd 算法，PPT 代码拼装 |
| `WarehouseSystem.h` | 跨结构协调（头文件） |
| `WarehouseSystem.cpp` | 跨结构同步，串联全员模块 |

**关键任务：**

- **第一天**写完所有 `.h` 文件，函数签名全组协商确定
- **每两天**检查菜鸟代码，D 优先（每步需要老手明确指示）
- **Week 3**负责最终集成调试

---

## 菜鸟 A：复制粘贴 + 体力活

| 文件 | 说明 |
|------|------|
| `BTree.h` | 从教材 `BTree 5.18/` 一字不改搬过来 |
| `BTree.cpp` | 同上，只改 `#include <stdlib.h>` → `#include <cstdlib>` |
| `SqList.h` | 从教材 `list 4.13/` 搬过来，删掉 `typedef int ElemType`，改用 `Goods` |
| `SqList.cpp` | 函数签名里 `ElemType` 全部换成 `Goods`，`DispList` 改成按 `Goods` 字段输出 |

**为什么给菜鸟：** 99% 复制粘贴，找对文件、改对签名即可。

---

## 菜鸟 B：纯循环函数

| 文件 | 说明 |
|------|------|
| `GoodsManager.h` | 声明（老手写好了，B 只需对着写 `.cpp`） |
| `GoodsManager.cpp` | 8 个函数，每个不超过 10 行 |

```
GM_Init        → 调 InitList
GM_FindPos     → for 循环 data[i].id == id
GM_GetById     → 调 GM_FindPos → 调 GetElem
GM_Add         → 查重 + 尾插
GM_Delete      → 找位置 + 删
GM_Update      → 找位置 + 覆盖
GM_Count       → 调 GetLength
GM_DisplayAll  → for 循环 printf
```

---

## 菜鸟 C：I/O + UI（代码量最大，无算法）

| 文件 | 说明 |
|------|------|
| `FileManager.h` | 声明 |
| `FileManager.cpp` | `fopen` → `fprintf`/`fgets` → `fclose` |
| `Menu.h` | 声明 |
| `Menu.cpp` | `printf` 菜单 + `scanf` + `switch`，约 150 行 |

**为什么给菜鸟 C（不是 D）：** 代码量最大但全是标准库函数调用，需要细心但不难。D 如果连 scanf/printf 都吃力，这份代码给他只会加剧拖延。

---

## 菜鸟 D：项目文档（最菜，代码最少）

### 代码部分（几乎没有）

| 文件 | 内容 |
|------|------|
| `main.cpp` | **就 3 行：** `#include "Menu.h"` + `int main() { Menu_Run(); return 0; }` |

**就这。** 全组最少的代码量。

### 文档部分（主力任务）

| 章节 | 内容 | 怎么让最菜的也能写 |
|------|------|---------------------|
| 1. 项目概述 | 系统功能、目标 | 老手口述 5 分钟，D 整理成文字 |
| 2. 数据结构设计 | SqList、BTree、MatGraph、CategoryTree 各一段 | 老手给 4 个结构体定义 + 一句话解释，D 扩成段落 |
| 3. 核心算法 | Dijkstra 最短路径 + BST 插入 | 老手给截图 + 伪代码，D 写文字描述 |
| 4. 模块架构 | 三层架构图 + 依赖关系 | 老手画草图，D 用 PPT/Word 做成正式图 |
| 5. 功能演示 | 10 个菜单功能的运行截图 | D 自己跑一遍，逐个截图 |
| 6. 测试用例 | 正常 + 边界情况 | 老手列 10 条测试项，D 执行、截图、记录结果 |
| 7. 总结收获 | 每人一段 | D 去问每个人"你最大的困难是啥"，录下来整理 |

### 关键：怎么帮 D 不卡住

| 编号 | 规则 |
|------|------|
| ① | **老手每步给明确指令**："今天你去做第 5 章功能演示，打开 exe，把 1 到 10 号菜单各截一张图"——而不是"你去写报告" |
| ② | **老手给模板**：不是给空白文档，而是给一个标题框架，D 往里头填 |
| ③ | **代码部分让 C 帮**：如果 main.cpp 那 3 行 D 都不会写，C 花 30 秒帮忙敲完 |
| ④ | **D 默认给老手看草稿**：每写完一章，老手花 5 分钟扫一眼，别等全写完才发现方向错了 |

## 依赖关系

```

  ├── 菜鸟 A（对照头文件搬教材）
```

## 时间线（3 周）

| 阶段 | 时间 | 老手 | 菜鸟 A | 菜鸟 B | 菜鸟 C | 菜鸟 D |
|------|------|------|--------|--------|--------|--------|
| Day 1 | 接口设计 | 写完全部 `.h` | 读接口 | 读接口 | 读接口 | 跟老手对一遍：这是什么项目、有哪些模块 |
| Week 1 | 各自开发 | CategoryTree | BTree+SqList | GoodsManager | FileManager | 写第 1-2 章 + 学会跑别人的测试 main |
| Week 2 初 | Review | 检查 A/B/C/D | 修 bug | 修 bug | 修 bug | 写第 3-4 章 |
| Week 2 末 | 集成 | WarehouseSystem | 联调 | 联调 | 联调 | 采访每人 5 分钟，收集素材 |
| Week 3 初 | UI | 集成调试 | 测试 | 测试 | 写 Menu | 跑全功能 → 截所有图 |
| Week 3 末 | 收尾 | 最终打包 | 修 bug | 修 bug | 修 bug | 写第 5-7 章 + 排版 |
| 最后 3 天 | 提交 | 终验 | — | — | — | 统稿 + 全组传阅 + 交 |

## 关键规则

| 规则 | 为什么 |
|------|--------|
| 老手第一天写完所有 `.h` | 菜鸟才有独立干活的基础 |
| 菜鸟各自编译验证 | 写临时 main 自测，不等到集成才爆 |
| 老手每两天 check 代码 | D 优先；A/B/C 抽查 |
| 头文件变更全组通知 | 否则联调类型不匹配 |
| C 的 Menu 最后一周写 | 等别人模块稳定 |
| **对 D 只说"做什么"，不说"你去写报告"** | 拆成具体任务：今天截 10 张图、今天写第 2 章 |
| **D 的代码只有 3 行 main.cpp** | 写不出来时 C 或老手直接帮掉，别卡住 |
